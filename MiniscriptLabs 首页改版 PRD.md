## 一、项目背景

当前 Miniscript 网站整体设计质量较高，但首页叙事方式和部分交互表达仍偏技术导向，导致普通用户在首次访问时较难快速理解：

- 这个产品解决什么问题
- 为什么值得关注
- 与自己有什么关系
- 应该从哪里开始理解和使用

本次改版的核心，不是增加更多技术说明，而是**调整叙事顺序与表达方式**，让网站先建立意义感与场景感，再承接技术解释与交互体验。

## 二、产品目标

### 2.1 总目标

将本网站从"面向开发者的技术说明页"优化为"普通用户也能理解价值、并愿意继续探索的产品介绍站点"。

### 2.2 具体目标

- 让用户快速理解 **Bitcoin 的花费条件可以被编程**
- 让用户理解 **Miniscript 的价值不只是技术实现，还包括表达清晰与钱包迁移能力**
- 让用户通过具体场景理解产品价值
- 让首页承担"建立认知"的职责
- 让 Applications 模块承担"场景说明"的职责

### 2.3 成功标准

- 用户进入首页后，可以在较短时间内理解产品的基本价值主张
- Applications 区域能帮助用户理解真实使用场景，而不是造成信息负担
- 首页整体阅读路径更顺畅，先易后难，先价值后技术

## 三、目标用户

### 3.1 核心目标用户

尝试自主保管资产、希望理解更复杂比特币花费条件的普通用户。

### 3.2 次级用户

- 对 Bitcoin script / Miniscript 有兴趣的进阶用户
- 希望理解迁移性、策略表达与应用场景的技术型用户

### 3.3 当前问题

当前页面部分内容默认用户已经具备较强技术背景，因此：

- 普通用户难以快速进入语境
- 首页前半段技术对比过早出现
- 用户在还未理解价值前，就被迫理解实现方式

## 四、核心产品判断

本次改版需要围绕以下三条核心信息展开：

1. **Bitcoin 的 UTXO 可以被编程**
2. **Miniscript 是一种更高级、更清晰、便于迁移的表达方式**
3. **这些能力能够映射到现实中的具体使用场景**

如果首页无法先把这三件事讲清楚，后续技术内容再完整，也会降低整体理解效率。

## 五、产品策略

### 5.1 首页策略

首页应从"技术解释优先"调整为"意义与场景优先，技术解释后置"。

### 5.2 Applications 策略

Applications 应继续保留，并提前到页面上半部分，用于帮助普通用户快速建立场景感。

## 六、信息架构调整方案

### 6.1 新的首页结构建议

推荐顺序如下：

1. 顶部主视觉与核心价值表达
2. 过渡模块
3. Applications
4. Core Concepts
5. Tech Stack / 技术说明

### 6.2 调整原则

- 页面上半部分优先服务普通用户理解
- 页面下半部分再逐步过渡到技术解释
- 每个模块承担明确职责，避免信息重复
- 将"为什么重要"放在"怎么实现"之前

## 七、详细需求

## 7.1 首页改版需求

### 需求 1：调整首页叙事顺序

**目标：** 先讲意义，再讲技术。

**现状问题：**

用户过早进入技术对比内容，尚未建立产品价值感知。

**需求说明：**

- 首页前半部分应优先说明比特币花费条件可编程这一核心概念
- 应优先展示具体应用场景和使用价值
- Core Concepts 与技术栈内容后置

**预期效果：**

让普通用户在继续阅读前，已经知道产品"是什么""为什么重要""适用于什么场景"。

### 需求 2：在 Challenge 之前增加过渡模块

**目标：** 搭建普通用户与技术概念之间的桥梁。

**需求说明：**

在标题区与 Challenge 模块之间新增一个过渡模块，建议结构为：

- 左侧展示单签名脚本与地址
- 右侧展示多签名脚本与地址
- 通过对比引导用户理解：地址背后对应脚本，脚本定义花费条件

**设计要求：**

- 重点是帮助用户建立概念，不是展开底层技术教学
- 文案应突出"从单签到多签，是理解更复杂条件的入口"
- 应强调"不止于多签钱包"的方向

**预期效果：**

用户在进入后续模块前，已经形成"脚本 → 地址 → 花费条件设计"的基本认知。

### 需求 3：强化首页文案主张

**目标：** 让首页价值主张更贴近普通用户。

**建议文案方向：**

- 比特币的花费条件不只是一把私钥
- 可以定义更复杂的支出规则
- 多签只是其中一种形式
- 更复杂的规则可以对应真实世界中的资产管理、继承、共管等场景

**要求：**

- 避免首页前部过度使用面向开发者的表达
- 让"可编程性""可迁移性""适用场景"成为主线

## 7.2 Applications 模块需求

### 需求 4：将 Applications 前置

**目标：** 更早建立场景理解。

**需求说明：**

- Applications 模块应移动到首页上半部分
- 放在过渡模块之后，Core Concepts 之前

**原因：**

Applications 对普通用户的帮助高于技术细节说明，更适合承担"价值说明"职责。

### 需求 5：对 Applications 做信息减法

**目标：** 降低信息密度，减少重复解释。

**保留内容：**

- 场景描述
- Policy 表达
- "上手试试"入口

**移除或弱化内容：**

- 右侧编译过程展示
- Miniscript 的详细展开
- Bitcoin script 的详细展开
- 重复性的脚本到地址过程展示

**原因：**

如果用户已在前面的过渡模块建立了"脚本到地址"的基础认知，就没有必要在每个场景卡片里继续重复完整技术链路。

### 需求 6：Applications 与 Playground 的承接

**目标：** 建立更清晰的页面分工。

**需求说明：**

- 每个场景卡片都保留"上手试试"入口
- 点击后直接进入 Playground 的对应场景
- 首页负责解释用途
- Playground 负责交互体验

## 八、优先级排序

### P0

- 首页叙事顺序调整
- Applications 前置
- Challenge 前新增过渡模块

### P1

- Applications 信息减法

### P2

- 文案进一步打磨

## 九、页面职责划分

### 首页

负责回答：

- 这是什么
- 为什么重要
- 能解决哪些真实问题

### Applications

负责回答：

- 它适合什么场景
- 每个场景大致如何表达
- 我可以从哪里开始尝试

## 十、风险与注意事项

- 若首页继续保留过多技术细节，可能削弱普通用户的进入体验
- 若 Applications 简化过度，可能导致产品可信度下降，因此仍需保留必要 policy 信息
