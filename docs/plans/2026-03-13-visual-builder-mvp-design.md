# 可视化构建 MVP 设计文档

> 状态：已确认  
> 日期：2026-03-13  
> 适用范围：`自己动手` 卡片对应的 Playground 新功能  
> 关联文档：`SPEC.md`、`agent.md`

## 1. 文档目的

本文档定义 `自己动手` 卡片在 MVP 阶段对应的完整产品方案。

该功能的目标不是再做一套“文本 IDE”，而是把 `自己动手` 变成一个真正可用、可惊艳、可扩展的 **可视化构建** 入口：

- 用户可以像搭积木一样构建 Bitcoin 花费策略
- 构建过程实时同步到 Policy 编辑器
- 当前条件下的满足状态直接叠加在构建画布上
- 右栏继续承担“路径枚举与技术结果”的职责
- 当前 MVP 只做 `可视化构建`
- 未来预留 `引导搭建` 模式，但不在本次实现范围内

本文档是该功能的单一事实源。若后续实现与旧讨论记录不一致，以本文档为准。

## 2. 核心结论

### 2.1 功能命名

- 未来双模式命名：
  - `引导搭建`
  - `可视化构建`
- 当前 MVP 只实现：`可视化构建`

### 2.2 入口定义

- 左栏场景选择器中，第一个位置的 `自己动手` 卡片不再是 Coming Soon 占位
- 点击后进入 `playgroundMode: 'build'`
- `build` 模式是独立的进阶入口，不替代现有场景模式

### 2.3 产品定位

`可视化构建` 不是自由流程图编辑器，也不是完全空白的节点画布。

它应被定义为：

**受约束的策略树编辑器**

特征如下：

- 用户编辑的是策略结构，不是随意拖线
- 系统负责布局，用户负责表达花费逻辑
- 画布既是编辑界面，也是状态可视化界面
- 不再在中栏额外放第二张完整“花费路径图”

### 2.4 关键设计决定

1. `build` 模式只有一张主画布
2. 主画布默认就是“构建视图”
3. 节点直接叠加 `已满足 / 等待 / 缺失` 状态
4. 不再提供单独的“构建图 / 路径图”切换
5. 节点参数编辑使用“节点旁小浮层”
6. 用户手改 Policy 时，若结构仍在支持范围内，画布回写
7. 用户写出当前构建器不支持的结构时，进入“文本主导模式”，而不是破坏构建器
8. 用户从 `自己动手` 主动进入 `build` 模式时，必须进入一个全新的空白构建工作区，不继承当前场景的 Policy
9. 数据模型从第一天就预留 `after` 和哈希锁，即使 MVP UI 不开放

## 3. 用户与成功标准

### 3.1 目标用户

该功能服务的是进阶用户，而不是第一次接触产品的纯新手：

- 已被预设场景教育过的学习者
- 想自己拼装策略的实践者
- 对 `policy / multisig / timelock` 已有初步概念的用户

### 3.2 成功标准

用户第一次进入 `自己动手` 后，应当能够在几分钟内完成以下事情：

- 从画布上的根占位选择策略类型，再逐步搭建自己的策略
- 看懂每个块代表什么花费条件
- 清楚知道哪些条件当前已满足、哪些仍缺失
- 在不手写 Policy 的情况下，搭出一个可编译的策略
- 在需要时切到 Policy 编辑器进行精修
- 理解“我改的是同一件事”，而不是两套彼此分离的 UI

## 4. MVP 范围

### 4.1 本次实现

第一版 MVP 支持：

- `自己动手` 卡片进入 `build` 模式
- 可视化策略树画布
- 根策略类型（占位节点上选择）：单签名、门限多签、AND 组、OR 组（**不再**通过 Policy 工具栏「模板」加载）
- 节点类型：
  - 签名
  - 全部满足
  - 任选其一
  - 阈值组 `k-of-n`
  - 相对时间锁 `older(blocks)`
- 角色主要来自左栏 Key 变量
- 节点浮层里允许轻量新建/重命名角色
- 可视化操作实时同步到 Policy 编辑器
- Policy 编辑器文本修改反向回写到构建器
- 画布节点叠加满足状态
- 右栏路径卡片点击后高亮对应分支
- 现有右栏技术结果、条件开关、时间滑块、状态横幅继续复用

### 4.2 明确不在 MVP

以下内容不在首版范围内：

- `引导搭建` 模式
- 自由拖线
- 自由摆放节点
- `after()` 绝对时间锁编辑
- 哈希锁编辑
- wrapper 级别的 Miniscript 可视化编辑
- 对所有合法 Policy 结构的完整图形回写支持
- 移动端完整构建体验

## 5. 模式与信息架构

### 5.1 Playground 顶层模式

Playground 维持两个顶层模式：

- `scenario`
  - 当前已有模式
  - 预设场景 + Policy 编辑 + 路径图
- `build`
  - 本次新增模式
  - 可视化构建 + Policy 同步 + 状态叠加

### 5.2 未来模式规划

未来当 `引导搭建` 实现后，可以在 `build` 模式内部再细分：

- `引导搭建`
- `可视化构建`

但当前 MVP 不需要在 UI 中提前做这个切换器，也不需要为用户展示未开放选项。

### 5.3 三栏结构

#### 左栏

保留现有三段结构：

- 场景/入口选择
- 角色与 Key 变量
- Context / Network

其中第一张卡片改为可点击的 `自己动手` 入口：

- 点击后切换到 `build` 模式
- 点击后应重置为一个新的空白构建工作区：
  - `policy` 置空
  - `strategyTree` 置空
  - `keyVariables` 置空
  - 清除当前场景编译结果、路径结果与条件模拟状态
  - 显示根占位节点（虚线「选择策略类型」）
- 不继承用户刚才所在场景的 Policy、路径图或角色变量
- 该卡片在激活态应与其他场景卡片一样有明确选中状态
- 文案仍保留“自由组合花费条件，设计属于你的策略”这一方向

补充规则：

- 上述“清空进入”只适用于用户主动点击 `自己动手`
- 若用户恢复的是一个已保存的 `build` 会话，或打开的是 `build` 模式分享链接，则允许带着已有 Policy / builder tree 直接进入

#### 中栏

`build` 模式下中栏结构为：

1. 顶部：Policy 编辑器
2. 中部：可视化构建画布
3. 底部：状态横幅 + 条件开关 + 时间滑块

重点变化：

- 原本“路径图区域”在 `build` 模式中被“可编辑构建画布”替代
- 不额外再放一张完整的花费路径图
- 画布本身通过状态叠加承担“可视化解释”职责

#### 右栏

右栏职责不变，继续作为：

- 路径卡片列表
- 技术结果面板

这样可以保持角色清晰：

- 中栏：搭建与理解
- 右栏：枚举与输出

## 6. 主画布设计

### 6.1 一张画布，两种能力

`build` 模式中栏只有一张主画布，但它同时承担两件事：

- 策略编辑
- 状态反馈

画布不是纯展示图，而是可操作的策略树。

### 6.2 为什么不再单独显示一张路径图

原因如下：

- 构建器和路径图在视觉上会高度相似
- 两张相似图并存会增加认知负担
- 用户在 `build` 模式中只想处理一件事：搭建策略

因此本设计采用：

**“同一张图，默认可编辑；状态直接叠加在节点上。”**

### 6.3 布局原则

画布应保持“思维导图 / 策略树”观感，而不是通用流程图风格：

- 自动布局
- 明确层级
- 从上到下或从左到右保持一致
- 不允许用户自由拖动节点改变结构语义
- 不允许用户自由连接任意两个节点

### 6.4 节点状态

节点继续复用现有语义状态体系：

- `satisfied`
- `pending`
- `missing`

但在 `build` 模式中，这些状态的优先级低于“编辑可读性”。

要求：

- 节点先看起来像可编辑积木
- 状态只作为第二层视觉信号叠加
- 避免状态装饰把整张图染得过重

### 6.5 路径联动

右栏点击某条路径时：

- 画布高亮该路径相关节点和边
- 非相关节点降低强调度，但不完��隐藏
- 用户可以在“全局结构”与“具体可花费路径”之间建立联系

## 7. 构建交互

### 7.1 初始状态：自由搭建起点

首次进入 `build` 模式时，画布显示一个**占位根节点**，而非模板选择器：

- 用户看到一个虚线边框的"选择策略类型"占位节点
- 点击后可选择：
  - 单签名（直接创建 pk 节点）
  - 都需要（创建 AND 组）
  - 任选一（创建 OR 组）
  - 门限多签（创建 threshold 组，默认 2-of-3）

这种设计的优势：

- 更自由，用户不受预设模板限制
- 更直观，用户从第一步就在"搭建"
- 更灵活，可以从任意结构开始

### 7.2 组节点的子条件添加

当用户选择组类型后，组节点下方显示一个**永久可见的"+ 添加条件"占位节点**。

点击"添加条件"后的选项：

- 添加签名
- 添加时间锁
- 添加嵌套组（再选择都需要/任选一/门限）

### 7.3 角色选择机制

添加签名时：

- 首先显示现有角色列表供选择
- 底部有"新建角色"选项（一键创建，与左栏角色「添加」同一套规则；自动生成测试公钥并选中到当前签名）

### 7.4 门限组的默认状态

选择"门限多签"时：

- 默认 k 值为 2
- 默认添加 3 个签名占位槽位，形成 2-of-3 结构
- 用户可以编辑每个槽位的角色
- 用户可以增删槽位和调整 k 值

### 7.5 添加条件占位的持久性

组节点下的"+ 添加条件"占位**始终显示**，用户可以无限添加子条件。

### 7.6 删除根节点的行为

如果用户删除了根节点（整棵树）：

- 弹出确认对话框
- 确认后重置为初始占位状态

### 7.7 节点编辑方式

节点编辑统一采用：

**节点旁小浮层**

不使用侧边详情面板，不在右栏编辑节点属性。

原因：

- 视线停留在当前语义上下文
- 更像搭积木
- 减少在中栏和右栏之间来回跳转

### 7.8 节点浮层内容

#### 签名节点

- 选择角色
- 快速新建角色
- 快速重命名当前角色
- 删除该节点

#### 阈值节点

- 编辑 `k`
- 编辑参与人数 `n`
- 为每个签名槽位分配角色
- 删除某个签名槽位
- 增加一个签名槽位

#### 相对时间锁节点

- 编辑区块数
- 显示人类可读时间
- 删除该节点

#### 组节点

- 添加子条件
- 包裹为 `任选其一`
- 包裹为 `全部满足`
- 删除该分支

### 7.9 明确不支持的交互

首版不支持：

- 拖拽连线
- 节点任意拖放重排
- 从边上直接插入复杂表达式
- 在画布里做完整角色管理表单

## 8. 角色与 Key 变量

角色管理仍以左栏为主。

原因：

- 角色变量属于全局资源
- 公钥映射也属于全局配置
- 若在画布内完整管理角色，会破坏画布聚焦

因此角色能力分层为：

- 左栏：完整角色管理、公钥映射、恢复默认、随机生成测试 key
- 节点浮层：轻量新建/重命名入口

画布里的角色选择器应直接读取左栏当前 Key 变量列表。

## 9. 数据模型

### 9.1 核心原则

`build` 模式不能直接把 Policy 文本当成唯一事实源。

应新增一个内部策略树模型作为构建器事实源，再将其序列化为 Policy。

### 9.2 推荐数据结构

```ts
type StrategyNode =
  | {
      id: string;
      kind: 'placeholder';
      placeholderType: 'root' | 'child';
    }
  | {
      id: string;
      kind: 'group';
      op: 'all' | 'any' | 'threshold';
      threshold?: number;
      children: StrategyNode[];
    }
  | {
      id: string;
      kind: 'signature';
      roleId: string;
    }
  | {
      id: string;
      kind: 'timelock';
      mode: 'relative' | 'absolute';
      value: number;
      unit: 'blocks' | 'date' | 'timestamp';
    }
  | {
      id: string;
      kind: 'hashlock';
      hashType: 'sha256' | 'hash256' | 'ripemd160' | 'hash160';
      digest: string;
    };
```

新增的 `placeholder` 节点用于：

- `root` 类型：初始占位根节点，用户选择策略类型后转换为真实节点
- `child` 类型：组节点下的"添加条件"占位，序列化时被忽略

### 9.3 为什么这样建模

这个模型比直接围绕 `pk()/older()/after()/sha256()` 建模更稳定：

- `older` 与 `after` 统一落在 `timelock`
- 哈希锁作为独立叶子节点预留
- 组合关系独立于具体 Policy 写法
- 后续可支持更多“视觉友好、语法不同”的表达方式

### 9.4 MVP 里实际开放的节点

首版 UI 只开放以下子集：

- `group(all)`
- `group(any)`
- `group(threshold)`
- `signature`
- `timelock(relative)`

以下结构只保留在数据模型里，不开放 UI：

- `timelock(absolute)`
- `hashlock`

## 10. 状态与同步规则

### 10.1 同步目标

系统必须始终尽量维持以下三个对象一致：

- `strategyTree`
- `policy`
- 编译结果 / 花费路径 / 节点状态

### 10.2 Builder -> Policy

当用户在构建器中编辑时：

1. 更新 `strategyTree`
2. 将 `strategyTree` 序列化为 Policy
3. 写入现有 `policy`
4. 走现有自动编译链路

### 10.3 Policy -> Builder

当用户手改 Policy 时：

1. 先尝试将 Policy 反向解析为 `strategyTree`
2. 若解析成功，且结构在构建器支持范围内：
   - 更新画布
   - 保持可视化编辑能力
3. 若解析成功，但包含当前构建器不支持的结构：
   - 不破坏已有 Policy
   - 进入“文本主导模式”
4. 若 Policy 语法错误：
   - 保持最后一次成功同步的画布
   - 显示错误与失同步提示

### 10.4 推荐状态机

建议为构建器新增同步状态：

```ts
type BuilderSyncState =
  | 'synced'
  | 'text-led'
  | 'compile-error';
```

含义如下：

- `synced`
  - 构建器与 Policy 双向可编辑
- `text-led`
  - Policy 有效，但超出当前构建器支持范围
  - 画布切为只读说明态
- `compile-error`
  - Policy 当前有错误
  - 画布保留最近一次成功结构

### 10.5 文本主导模式

这是未来扩展能力的关键保障。

当用户写出当前构建器不支持的 Policy 结构时：

- 不丢失用户输入
- 不强行“猜错”后回写成错误画布
- 不把不支持的结构默默删掉

界面行为：

- 中栏保留画布区域
- 显示明确提示：
  - 当前 Policy 可以继续编译和查看结果
  - 但其中包含当前可视化构建器暂不支持的结构
- 提供回到可视化支持范围的提示动作

用户已明确接受该策略。

## 11. 时间与条件模拟

### 11.1 MVP 现状

当前 Playground 的时间滑块实际主要服务 `older()`。

`after()` 的完整模拟当前并不成立，因此 MVP 不开放 `after` 编辑是正确取舍。

### 11.2 本次要求

即使 MVP 只支持 `older`，实现时也必须避免把时间模拟写死为“相对时间锁专用”。

建议抽象为统一的时间上下文能力，为未来保留：

- 当前链高度
- 当前日期/时间
- UTXO 经过的相对区块数

### 11.3 条件模拟区

底部区域继续保留：

- 条件开关
- 时间滑块
- 状态横幅

差异在于：

- 节点状态直接跟随这些控制器变化
- 用户无需再看另一张路径图才知道当前状态

## 12. 未来扩展位：`after` 与哈希锁

### 12.1 设计约束

当前设计必须明确为未来扩展负责。

即：

- 不能把叶子节点类型写死为 `signature | older`
- 不能把时间滑块写死为“只表示 elapsed blocks”
- 不能把构建器序列化逻辑写成只认识 `pk / older / and / or / thresh`

### 12.2 `after()` 的未来接入

未来接入 `after()` 时，预期会增加：

- `timelock(absolute)` 节点 UI
- 绝对区块高度/日期输入
- 更完整的时间模拟上下文
- 路径可满足判断中对 `timelock_absolute` 的真实计算

如果当前模型按本文档设计，则 `after()` 属于中等规模增量，而不是重构。

### 12.3 哈希锁的未来接入

未来接入哈希锁时，预期会增加：

- `hashlock` 节点 UI
- 哈希类型选择
- digest 输入或示例值生成
- 更人话的条件模拟交互，例如 “Reveal Secret”

如果当前模型按本文档设计，则哈希锁属于低到中等规模增量。

  - 构建器支持范围判断
  - 节点操作工具函数

### 13.3 与现有 flow 的关系

建议不要把可视化构建器直接硬塞进当前 `tree-to-flow.ts` 的只读路径图逻辑。

更合理的方向是：

- 保留现有 flow 作为“编译后路径可视化”基础设施
- 为 `build` 模式新增更贴近编辑语义的 builder 层
- 在需要时复用现有配色、节点状态���局部布局经验

## 14. MVP 验收标准

以下条件同时满足，方可视为 MVP 达标：

1. 用户点击 `自己动手` 后可进入 `build` 模式
2. 用户看到占位根节点，可以选择策略类型开始自由搭建
3. 用户可通过可视化操作搭出至少以下结构：
   - 单签
   - 2-of-3 多签
   - `and(pk(User),or(pk(Service),older(...)))` 类恢复结构
4. 组节点下始终显示"添加条件"占位，支持持续扩展
5. 构建器修改会实时同步到 Policy 编辑器
6. 用户手改支持范围内的 Policy，画布会同步更新
7. 用户手改出不支持结构时，系统进入文本主导模式，不丢输入
8. 节点状态会随着签名开关和时间滑块变化
9. 右栏点选路径能在画布中形成高亮反馈
10. 整个功能不破坏现有场景模式
11. 实现方式对未来 `after` 和哈希锁保持兼容

## 15. 延后事项

以下事项已明确，但延后到后续版本：

- `引导搭建`
- `after()` 绝对时间锁的完整编辑与模拟
- 哈希锁节点编辑
- 更复杂 fragment 的可视化支持
- 移动端构建体验
- 真正的“高级模式”分层入口

## 16. 最终定义

本 MVP 的本质不是“给 Policy 编辑器加一张图”，而是：

**把 `自己动手` 变成一个以策略树为核心、以文本为同步视图、以状态叠加为解释层的可视化构建入口。**

如果实现偏离了这一定义，例如重新做成自由拖线流程图、双图并列展示、或仍让 Policy 文本作为唯一事实源，则视为偏离本设计目标。
