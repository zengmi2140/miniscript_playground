# 文档工程审计报告（2025-06-12）

> **审计范围**：ScriptWise 项目 Harness 文档体系（AGENTS.md、docs/PRODUCT.md、docs/ARCHITECTURE.md、docs/DESIGN.md、TASKS.md、README.md）。
> **审计目标**：从 Harness 工程（文档作为 AI Agent 开发治具）角度，识别改进空间。
> **使用方式**：本文是只读参考文档，承载完整问题分析与建议方案。`TASKS.md` 通过锚点链接引用本文各章节。

---

## 1. README 与 AGENTS.md / ARCHITECTURE.md 存在大量重复且版本不一致

**问题**

`README.md` 写的技术栈是 "Next.js 14 + React 18"，而 `ARCHITECTURE.md` 和 `AGENTS.md` 已更新为 "Next.js 15 + React 19"。项目结构部分也有简化版与完整版两份维护。Roadmap 与任务交接文档的已完成列表无关联。

**影响**：Agent 读到不同版本会产生困惑；外部贡献者看到过期信息。

**建议**

`README.md` 定位为"面向外部开发者/贡献者的入口"，技术细节用链接指向 `ARCHITECTURE.md`，避免同一事实两处维护。版本号只在一个地方声明（`package.json`），其余用引用。

**涉及文件**：`README.md`、`docs/ARCHITECTURE.md`

---

## 2. AGENTS.md 的"开始前必须阅读"列表缺少条件分支

**问题**

当前写法是"每次都必须读 4 个文件"。对于小改动（比如改一个 i18n 字符串），强制读 `ARCHITECTURE.md` 全篇是浪费 token。没有按任务类型给出"最小阅读集"。

**建议**

增加一个"按改动类型的最小阅读矩阵"，例如：

| 改动类型 | 最小阅读集 |
|----------|-----------|
| i18n 文案 | AGENTS.md + i18n 文件 |
| 样式微调 | AGENTS.md + DESIGN.md |
| 引擎 / 编译 | AGENTS.md + ARCHITECTURE.md |
| 路由 / 产品范围 | AGENTS.md + PRODUCT.md |
| 通用功能 | 全部 4 个文件 |

**涉及文件**：`AGENTS.md`

---

## 3. TASKS.md 的"已完成"列表过长且没有时间线

**问题**

当前"已完成"区域堆积了所有历史工作，但没有日期、没有版本号，无法判断"最近一次改动是什么"。对 Agent 来说，最重要的信息是"上一轮做了什么、改了哪些文件、验证了什么"，而不是全部历史。

**建议**

- 将"已完成"改为按轮次/日期分组（例如 `## 2025-06 轮次`）。
- 只保留最近 2-3 轮的详细记录，更早的归档到折叠区。
- 增加一个 `## 最近改动摘要` 区域，用 3-5 行概括最后一次 session 的核心产出。

**涉及文件**：`TASKS.md`

---

## 4. AGENTS.md 缺少"不需要读文档"的快捷通道

**问题**

对于纯样式微调、修复 typo、更新测试快照等低风险操作，当前规则是"必须先读 4 个文件"。这会导致 Agent 在简单任务上浪费大量时间。

**建议**

增加一个 "Quick Tasks" 例外清单，明确哪些类型的改动可以跳过全文阅读：

- i18n 文案修改（仅改 `zh.ts` / `en.ts`）
- CSS / Tailwind 类名微调
- 测试快照更新
- 文档本身的 typo 修复
- 依赖版本升级（仅 `package.json` + `package-lock.json`）

**涉及文件**：`AGENTS.md`

---

## 5. "完成标准"缺少量化验收条件

**问题**

当前完成标准是定性描述（"按预期工作"、"已实际验证"），但没有定义：

- 测试覆盖率的最低要求是多少？
- lint 0 warning 还是 0 error？
- 构建必须成功（`npm run build`）吗？

**建议**

将完成标准具体化：

- `npm run lint` → 0 error（warning 不阻塞）
- `npm run typecheck` → 0 error
- `npm run test` → 全绿
- `npm run build` → 成功（可选，视改动范围）

**涉及文件**：`AGENTS.md`

---

## 6. 缺少"文档同步"的具体触发规则

**问题**

`AGENTS.md` 说"改动落地后同步更新文档"，但没有给出判断标准——Agent 如何知道一个改动是否影响了 PRODUCT.md 的范围？

**建议**

增加一个具体的"触发矩阵"：

| 改动类型 | 需更新的文档 |
|----------|-------------|
| 新增 / 删除路由 | PRODUCT.md |
| 新增 / 删除 `src/lib/` 子目录 | ARCHITECTURE.md |
| 新增 / 修改颜色变量或字号 | DESIGN.md |
| 新增 / 修改预设场景 | PRODUCT.md「预设场景」段 |
| 仅修改组件内部实现、不改变对外接口 | 无需更新文档 |

**涉及文件**：`AGENTS.md`

---

## 7. ARCHITECTURE.md 混合了"是什么"和"怎么改"

**问题**

"目录结构"、"核心链路"是描述性内容（what）；"常见改动入口"是操作性内容（how）。两者混在一起，Agent 做小改动时被迫扫描全篇。

**建议**

将"常见改动入口"表独立为 `## Quick Reference: 改动入口速查` 节，放在文件顶部（紧接技术栈之后），方便 Agent 快速定位。

**涉及文件**：`docs/ARCHITECTURE.md`

---

## 8. DESIGN.md 缺少与代码的双向映射

**问题**

`DESIGN.md` 定义了 CSS 变量（如 `--bg-base`、`--semantic-key`），但没有说明这些变量在代码中通过什么方式引用（Tailwind 类名？CSS `var()`？`tailwind.config.js` theme extend？）。Agent 新增组件时不知道该用 `bg-bg-base` 还是 `bg-[var(--bg-base)]`。

**建议**

增加一个"Token 使用方式"段落，说明设计 token 是通过 `tailwind.config.js` 的 `theme.extend.colors` 映射为 Tailwind 工具类，还是直接在 CSS 中使用 `var()`。给出一个标准的组件样式示例。

**涉及文件**：`docs/DESIGN.md`

---

## 9. 缺少"约定与反模式"文档

**问题**

项目有很多隐含约定散落在 `ARCHITECTURE.md` 各处，例如：

- 导入 Bitcoin 库时必须用 dist 子路径
- 引擎层不引用 React
- 组件不直接读 `playground-store`，而是通过 hook
- 新增 i18n key 必须中英文同时添加

Agent 容易在不了解这些约定的情况下写出"看起来正确但违反项目规范"的代码。

**建议**

在 `AGENTS.md` 或 `ARCHITECTURE.md` 中增加一个 `## Conventions & Anti-patterns` 段落，集中列出项目的编码约定。

**涉及文件**：`AGENTS.md` 或 `docs/ARCHITECTURE.md`

---

## 10. 文档之间缺少"单一事实源"原则的严格执行

**问题**

- 技术栈版本号出现在三个地方：`README.md`（已过期）、`ARCHITECTURE.md`、`package.json`。
- "不允许做的事"在 `AGENTS.md` 和 `ARCHITECTURE.md` 都有，但措辞略有不同。

**建议**

确立"单一事实源 + 引用"原则：

- 版本号只在 `package.json` 中定义，文档中说"以 `package.json` 为准"。
- 硬边界只在 `AGENTS.md` 中完整列出，`ARCHITECTURE.md` 用引用链接。

**涉及文件**：`README.md`、`docs/ARCHITECTURE.md`、`AGENTS.md`

---

## 11. 改进建议.md 放在仓库根目录但不在 docs/ 下 ✅ 已解决

**问题**

`改进建议.md` 是产品级的内容规划文档，但放在了根目录，且文件名是中文。如果项目后续有更多此类文档，根目录会变得混乱。

**处理结果**

已移至 `docs/CONTENT-ROADMAP.md`，原根目录文件已删除。

**涉及文件**：`docs/CONTENT-ROADMAP.md`（原 `改进建议.md`）

---

## 12. 缺少"文档健康度自检"机制 ✅ 已解决

**问题**

没有任何自动化手段检测文档是否与代码同步。随着时间推移，文档腐烂是必然的。

**处理结果**

- 新增 `scripts/check-doc-health.mjs`，扫描根目录与 `docs/` 下的 Markdown 文件，检查内部引用的文件路径是否存在。
- 已注册为 npm script：`npm run doc:health`。
- 建议在 CI 中运行此脚本作为文档质量门禁。

**涉及文件**：`scripts/check-doc-health.mjs`、`package.json`
